## Resumen ejecutivo
- El diálogo (`asignar-tasador-dialog.tsx`) ya era cmdk-based con orden por cobertura, carga, alerta fuera de cobertura, nota y confirmación con consecuencias.
- P7 cableó la asignación al endpoint real `POST /api/solicitudes/[id]/asignar` y completó la confirmación (§8.1).
- `pnpm tsc --noEmit` y `pnpm build` limpios (exit 0).

## Decisiones técnicas
- **Cableado real de la asignación:** `handleConfirmado` (en `solicitud-detail.tsx`) ahora es `async`, dispara `POST /api/solicitudes/[id]/asignar` con `{tasadorId, motivo}` y aplica el estado optimista sólo si el POST responde ok (incl. `pendiente_make`). En error → toast §6 canónico de red, sin aplicar cambios.
- **Guarda de record-id real** (`/^rec[a-zA-Z0-9]{14}$/`): en `/consola` (ids reales) dispara el POST; en la demo mock `/` (ids no-record) aplica solo estado local, evitando un 404 que rompería la demo.
- **No `router.refresh()` todavía:** como la persistencia está mockeada (`pendiente_make`), refrescar traería el registro sin cambios y borraría el update optimista. Se difiere a P9 (cuando Make persista de verdad).
- **Toast §6 canónico:** "Solicitud asignada a {nombre}" (se quitó el punto final que tenía).
- **Confirmación completada (§8.1 paso 2):** se agregó la consecuencia "Se enviará el correo de asignación al tasador" (faltaba); ahora lista las 4.
- `onConfirmado` pasa `(tasadorId, nombre, nota)` — antes solo el nombre; el id se necesita para el POST.

## Decisión de datos (candidatos) — mantener mock, deuda H-05
- El picker sigue usando el mock `TASADORES` (con `cobertura`/`carga`/`capacidad`/`rut`) en vez de `/api/tasadores/candidatos`, porque `M_Tasadores` **no tiene** `zonas_cobertura` ni `casos_en_curso` (H-05). El endpoint real devuelve `coberturaDisponible:false` y solo `{id,nombre,capacidadActiva}` — cambiar ahora degradaría la UI (sin orden por cobertura ni barras de carga). Swap pendiente cuando existan esos campos.

## Overrides / notas
- Se mantuvo el diálogo como archivo único (no la carpeta `dialogo-asignacion/` del plan).
- El estado del detalle sigue siendo optimista/local; la persistencia real depende de Make (P9). El POST se dispara de verdad, pero hoy retorna `pendiente_make`.
