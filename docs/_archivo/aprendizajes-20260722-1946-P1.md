## Decisiones técnicas
- **Uniones derivadas con `(typeof ARRAY)[number]`** sobre los arrays de labels ya presentes (`TIPOS_BIEN`, `ROLES_CONTACTO_VISITA`, etc.), en vez de reescribir los valores a slugs como propone el plan §2.1. Cero cambio de valores → cero ruptura de list/detail/form.
- **Campos nuevos de `Solicitud` como opcionales** (`ejecFormalizador?`, `modoCreacion?`, `tipoClienteOrigen?`, `origenDireccion?`, `emailThreadId?`, `nivelSla?`) para no invalidar los objetos mock de `SOLICITUDES`.
- **Entidades siguen con `string`** en campos como `Unidad.tipoBien` o `ContactoVisita.rol`; las uniones quedan exportadas y disponibles, pero adoptarlas en las entidades se difiere a P4/P5 para no romper el mapeo del formulario (que produce `string` vía zod).

## Overrides aplicados (rutas reales vs plan)
- Plan §2.1 pedía archivos `lib/types/*.ts` + barrel `index.ts` → se extendió `lib/console-data.ts` (fuente única existente). No se creó `lib/types/`.
- Plan usaba snake_case y valores-slug → se respetó camelCase + valores-label del repo.
- Plan `TipoPropiedad = nuevo|usado` colisiona con el campo `Solicitud.tipoPropiedad` (Casa/Depto…) → se mantuvo `NuevoUsado` como el tipo nuevo/usado; no se creó alias `TipoPropiedad` para evitar ambigüedad.
